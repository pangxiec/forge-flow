# AI 全链路开发平台第一阶段需求沉淀

## 1. 项目定位

面向企业研发团队建设一个 AI 全链路开发平台，帮助团队标准化完成从需求提交、PRD 生成、原型确认、架构设计、代码生成、测试用例生成到 Git 交付的全过程。

平台目标不是替代研发团队，而是把企业内部的研发规范、组件规范、架构模板、代码模板和审批流程沉淀到统一平台中，让 AI 在受控流程下辅助产品、业务、设计、开发、测试和架构角色协同交付。

第一阶段开发设计详见 `AI全链路开发平台第一阶段开发设计.md`。

## 2. 核心价值

1. 标准化需求到交付流程，减少口头需求、散落文档和重复沟通。
2. 自动生成 PRD、原型、接口设计、数据库设计、代码和测试用例，提高研发前期产出效率。
3. 通过多人审批和冻结机制，降低 AI 生成内容直接进入开发造成的风险。
4. 将公司知识库、研发规范和代码模板接入 AI 生成链路，保证产物符合企业内部标准。
5. 自动生成开发包并推送到公司 Git，程序员可 pull 到本地后继续使用个人 AI 工具开发。

## 3. 第一阶段目标

第一阶段聚焦“从需求上传到 Git 交付开发包”的最小闭环：

```text
需求上传
↓
AI 生成 PRD
↓
多人确认
↓
AI 生成原型页面
↓
多人确认
↓
AI 生成开发包
↓
推送 Git
↓
程序员 pull 本地继续开发
```

第一阶段暂不追求完整自动上线，也不要求 AI 生成的代码达到生产可直接发布水平。生成代码的目标是可作为正式项目骨架继续开发，并且前后端代码必须通过基础编译和启动验证。重点是建立清晰流程、审批节点、产物结构和 Git 交付机制。

第一阶段仅服务公司内部研发团队，暂不支持多公司、多租户、多个项目空间、团队空间和权限隔离。这些能力纳入后续迭代规划。

## 4. 用户角色

| 角色 | 核心职责 |
| --- | --- |
| 超级管理员 | 维护系统配置、模型配置、Gitea 配置、角色权限、审计日志和全局知识库 |
| 项目经理 | 管理项目成员和阶段流转，对各阶段产物拥有最终确认权 |
| 产品经理 | 上传需求、确认 PRD、发起原型确认、管理需求版本 |
| 需求方 | 澄清业务规则、确认 PRD 和原型是否满足业务目标 |
| 架构师 | 确认架构方案、权限模型、技术选型和系统边界 |
| 前端负责人 | 确认原型交互、页面结构、前端工程规范和前端代码可继续开发性 |
| 后端负责人 | 确认接口、数据库、后端工程规范和后端代码可继续开发性 |
| 测试负责人 | 确认验收标准、测试用例和测试覆盖范围 |
| 普通开发 | 查看冻结产物和 Git 交付信息，pull 代码到本地继续开发 |

## 5. 全阶段能力地图

### 5.1 需求中心

用于收集原始需求，并通过 AI 完成初步结构化。

核心能力：

- 需求上传，支持文本、文档、图片、会议纪要等形式。
- AI 自动补全需求背景、目标用户、业务流程、边界条件和异常场景。
- AI 生成澄清问题，由产品经理或需求方补充。
- 需求版本管理，保留每次变更记录。
- 需求状态流转，例如草稿、待澄清、待确认、已确认、已废弃。

### 5.2 PRD 中心

用于将确认后的需求转化为正式产品需求文档。

核心能力：

- AI 根据需求生成完整 PRD。
- PRD 包含业务背景、目标、范围、用户角色、业务流程、功能清单、页面说明、规则说明、异常场景、验收标准。
- 支持多人评论、修改建议和问题追踪。
- 支持审批确认。
- 支持 PRD 冻结，冻结后作为后续原型、架构和代码生成的基准版本。
- PRD 冻结后允许变更，变更属于开发阶段的需求变更，应补充更新到需求文档，并使用颜色和日期标注。此类变更不重复触发代码生成，由开发人员在本地继续修改。

### 5.3 原型中心

用于根据 PRD 生成可交互前端原型，供需求方和开发负责人确认。

核心能力：

- AI 根据 PRD 生成 Vue3 + TypeScript + Element Plus 风格的页面原型。
- 支持页面路由、核心表单、列表、详情、审批弹窗、状态流转等交互。
- 需求方确认页面流程和字段是否符合预期。
- 前端负责人确认组件、布局和交互实现可行性。
- 后端负责人确认接口、数据和状态流转支撑可行性。
- 支持原型版本管理和确认记录。

### 5.4 架构中心

用于生成系统技术方案和工程设计。

核心能力：

- 生成数据库设计，包括表结构、字段、索引、关系和初始化数据。
- 生成接口设计，包括接口路径、请求参数、响应结构、错误码和权限要求。
- 生成技术方案，包括模块划分、服务边界、缓存策略、文件存储策略和异常处理。
- 生成权限模型，包括角色、菜单、按钮权限、数据权限和审批权限。
- 支持架构师审批确认。

### 5.5 代码中心

用于生成项目代码并推送到公司 Git。

核心能力：

- 生成前端工程代码。
- 生成后端 Spring Boot 工程代码。
- 生成数据库脚本。
- 生成基础测试代码。
- 生成 README、部署说明和开发说明。
- 根据企业代码模板生成统一工程结构。
- 接入公司后端代码结构规范，确保生成的 Spring Boot 工程符合内部目录、分层、命名和基础依赖约定。
- 自动执行前后端基础编译检查和启动验证。
- 通过 Gitea API 创建仓库、提交代码、创建分支或发起合并请求。
- Gitea 仓库命名由页面填写，不由系统强制自动推导。
- 分支名称支持页面填写，默认分支为 main。

### 5.6 测试中心

用于生成测试设计和自动化测试资产。

核心能力：

- 根据 PRD 和接口设计生成测试用例。
- 生成接口测试用例。
- 生成自动化测试脚本。
- 生成验收标准。
- 关联缺陷和需求版本。

### 5.7 发布中心

用于管理部署配置和发布记录。

核心能力：

- 生成 Dockerfile 和 docker-compose 配置。
- 生成 CI/CD 流水线配置。
- 管理开发、测试、预发、生产环境配置。
- 记录部署版本、部署时间、部署人和发布结果。

第一阶段可先生成基础 Docker 配置和部署说明，不强制实现完整发布流水线。

### 5.8 知识库中心

用于沉淀企业内部标准，作为 AI 生成内容的上下文来源。

核心能力：

- 公司开发规范。
- 前端组件规范。
- 后端项目结构规范，详见 `knowledge-base/后端项目结构规范.md`。
- 后端接口返回规范，详见 `knowledge-base/后端项目结构规范.md`。
- 数据库设计规范。
- 数据库表设计规范，详见 `knowledge-base/数据库表设计规范.md`。
- 代码模板。
- 安全规范。
- 测试规范。
- 业务领域知识。

知识库建议使用 Elasticsearch 做检索，并结合向量检索能力增强语义召回。

### 5.9 审批协同中心

用于管理多角色确认流程，确保 AI 生成产物进入下一阶段前经过人工确认。

审批流详细规则详见 `审批流规则.md`。

核心能力：

- 产品确认。
- 需求方确认。
- 前端确认。
- 后端确认。
- 测试确认。
- 架构师确认。
- 审批意见、驳回原因、修改记录。
- 审批模板和审批流程配置。
- 每个阶段必须审批通过后才能进入下一阶段。
- 审批驳回后由产品手动调整内容，再重新触发对应阶段生成。

第一阶段审批流：

```text
需求创建
↓
AI 分析
↓
产品确认
↓
需求方确认
↓
PRD 冻结
↓
AI 生成原型
↓
产品确认
↓
需求方确认
↓
前端确认
↓
后端确认
↓
原型冻结
↓
AI 生成开发包
↓
架构师确认
↓
Git 初始化
↓
开发开始
```

阶段审批人与确认职责：

| 阶段 | 审批人/确认人 | 说明 |
| --- | --- | --- |
| AI 分析后 | 产品经理 | 确认需求理解、缺失信息和澄清问题是否完整 |
| PRD 冻结前 | 产品经理、需求方 | 产品确认文档结构和范围，需求方确认业务目标和规则 |
| 原型冻结前 | 产品经理、需求方、前端负责人、后端负责人 | 产品确认页面表达，需求方确认业务流程，前端确认交互可实现，后端确认接口和数据支撑可实现 |
| 开发包生成后 | 架构师 | 确认架构设计、工程结构、接口、数据库和技术方案符合规范 |
| Git 初始化前 | 项目经理 | 对进入开发阶段进行最终确认 |

### 5.10 角色权限规范

第一阶段角色权限按平台管理、项目推进、产物确认、开发协作四类划分。

| 角色 | 权限范围 |
| --- | --- |
| 超级管理员 | 管理系统配置、模型配置、Gitea 配置、角色权限、审计日志和全局知识库 |
| 项目经理 | 创建和管理项目，配置项目成员，查看全部产物和审批记录，拥有各阶段最终确认权 |
| 产品经理 | 创建需求，上传需求材料，维护 PRD，处理审批驳回，发起 PRD 冻结和原型确认 |
| 需求方 | 查看需求、PRD 和原型，补充业务说明，确认业务规则、流程和验收目标 |
| 架构师 | 查看 PRD、原型和架构设计，确认技术方案、接口、数据库、工程结构和开发包 |
| 前端负责人 | 确认原型交互、页面结构、前端工程规范和前端代码可继续开发性 |
| 后端负责人 | 确认接口设计、数据库设计、后端工程结构和后端代码可继续开发性 |
| 测试负责人 | 查看 PRD、原型、接口设计和测试用例，确认验收标准和测试覆盖范围 |
| 普通开发 | 查看已冻结产物和 Git 交付信息，拉取代码后继续开发，不具备阶段审批和冻结权限 |

### 5.11 Agent 中心

用于管理不同阶段的 AI Agent。

核心 Agent：

- PRD Agent：负责需求分析、澄清问题和 PRD 生成。
- Prototype Agent：负责前端原型生成。
- Backend Agent：负责后端代码、接口和数据库生成。
- Frontend Agent：负责前端页面和组件代码生成。
- Test Agent：负责测试用例和测试脚本生成。
- Git Agent：负责仓库创建、代码提交和分支管理。

## 7. 第一阶段核心流程

### 7.1 需求到 PRD

1. 产品创建项目并上传原始需求。
2. 系统调用 PRD Agent 分析需求。
3. 系统生成需求摘要、缺失信息和澄清问题。
4. 产品经理确认需求理解和 PRD 生成输入。
5. 需求方确认业务目标、业务规则和澄清信息。
6. 系统生成 PRD 初稿。
7. 产品经理和需求方评论、修改并确认。
8. 审批通过后 PRD 冻结，项目经理拥有最终确认权。

### 7.2 PRD 到原型

1. 系统基于冻结 PRD 调用 Prototype Agent。
2. 生成前端原型页面。
3. 产品预览并调整页面说明。
4. 需求方确认业务流程。
5. 前端负责人确认页面结构、交互和组件实现可行性。
6. 后端负责人确认接口、数据和状态流转支撑可行性。
7. 原型审批通过后冻结，项目经理拥有最终确认权。

### 7.3 原型到开发包

1. 系统基于冻结 PRD 和冻结原型生成架构设计。
2. 架构师确认数据库、接口、模块设计和工程结构。
3. 系统生成前端代码、后端代码、测试代码和数据库脚本。
4. 系统生成 README 和本地启动说明。
5. 系统在沙箱中执行前后端基础编译检查和启动验证。
6. 项目经理确认进入 Git 初始化。
7. Git Agent 根据页面填写的仓库名称自动创建 Gitea 仓库。
8. Git Agent 根据页面填写的分支名称提交代码，默认分支为 main。
9. 程序员 pull 到本地继续开发。

## 13. 已确认问题与阶段结论

### 13.1 产品边界

1. 第一阶段只服务内部研发团队，不支持多公司、多租户。
2. 第一阶段暂不支持多个项目空间、团队空间和权限隔离，该能力放入后续迭代。
3. 平台生成代码的目标是可作为正式项目骨架继续开发。
4. 生成的前后端代码必须通过基础编译和启动验证。

### 13.2 流程规则

1. 每个阶段必须审批通过后才能进入下一阶段。
2. 项目经理角色拥有最终确认权。
3. 审批流详细规则详见 `审批流规则.md`。
4. PRD 冻结后允许变更，变更属于开发阶段中的需求变更，需要补充并更新到需求文档，使用颜色和日期进行标注。
5. PRD 冻结后的需求变更不重复触发代码生成，由开发人员在本地自行修改。
6. 审批驳回后由产品手动调整内容，再重新触发对应阶段生成。

### 13.3 AI 生成策略

1. 第一阶段允许调用公网大模型。
2. 上传给大模型的敏感数据必须先脱敏。
3. 平台操作需要完整审计日志。
4. 模型调用记录、成本统计、生成失败重试、人工编辑内容反向进入知识库、企业私有化模型或本地模型支持可在详细设计阶段继续拆分。

### 13.4 知识库

1. 第一阶段需要接入生成后端代码结构的公司规范，详见 `knowledge-base/后端项目结构规范.md`。
2. 知识库文档维护人、知识库文档审批机制、按项目/部门/技术栈隔离知识库等能力放入后续详细设计。

### 13.5 Git 集成

1. 公司最终使用 Gitea，第一阶段统一以 Gitea API 为主集成目标。
2. 由平台自动创建仓库。
3. Gitea 仓库名称需要在页面上填写。
4. 支持分支策略，分支名称需要在页面上填写，默认值为 main。
5. 是否自动创建 Pull Request、Git 凭证使用用户 Token 还是平台机器人账号，放入详细设计阶段确认。

### 13.6 安全与合规

1. 上传需求中可能包含敏感数据，推送给大模型前必须脱敏。
2. 第一阶段允许调用公网大模型。
3. 平台操作需要完整审计日志。
4. 敏感数据识别范围包括密钥配置、数据库连接、账号密码、Token、AccessKey、SecretKey、内部服务地址等。
5. 生成初始项目时，密钥配置、数据库连接等敏感配置默认为空，不生成真实值。
6. 生成代码安全扫描放入后续迭代。

### 13.7 工程质量

1. 生成代码需要自动执行编译检查。
2. 第一阶段需要在沙箱中进行前后端基础启动验证。
3. 后端代码结构必须符合公司规范。
4. 测试用例自动执行、完整代码风格检查、失败产物回滚放入后续详细设计或迭代规划。

## 14. 第一阶段建议里程碑

### M1：需求与 PRD 闭环

- 项目创建。
- 需求上传。
- AI 需求分析。
- 澄清问题生成。
- 产品确认。
- 需求方确认。
- PRD 生成、评论和审批。
- PRD 冻结。

### M2：原型与审批闭环

- 根据 PRD 生成前端原型。
- 原型在线预览。
- 产品确认。
- 需求方确认。
- 前端负责人确认。
- 后端负责人确认。
- 原型版本管理。
- 原型冻结。

### M3：架构与开发包生成

- 数据库设计生成。
- 接口设计生成。
- 前后端项目骨架生成。
- 接入公司后端代码结构规范。
- 前后端基础编译检查和启动验证。
- 测试用例生成。
- README 和启动说明生成。
- 架构师确认开发包。

### M4：Git 交付闭环

- Gitea API 集成。
- 平台自动创建仓库。
- Git 初始化前由项目经理最终确认。
- 页面填写 Gitea 仓库名称。
- 页面填写分支名称，默认值为 main。
- 提交开发包。
- 生成交付记录。
- 程序员 pull 本地继续开发。

## 15. 第一阶段验收标准

1. 产品可以上传一份原始需求，并看到 AI 生成的结构化分析结果。
2. 系统可以生成一份完整 PRD，并支持评论、修改、审批和冻结。
3. 系统可以基于冻结 PRD 生成可预览的前端原型。
4. 需求方、前端负责人和后端负责人可以对原型进行确认或驳回。
5. 系统可以生成数据库设计、接口设计、前端代码、后端代码、测试用例和说明文档。
6. 生成的前后端代码可以通过基础编译检查和启动验证。
7. 系统可以根据页面填写的仓库名称自动创建 Gitea 仓库，并按页面填写的分支名称推送完整开发包，分支默认值为 main。
8. 程序员可以 pull 仓库到本地，并根据 README 启动或继续开发。
9. 每个阶段的生成任务、审批记录和产物版本都有记录可追溯。
10. 平台操作具备完整审计日志。
11. 生成初始项目时，密钥配置、数据库连接等敏感配置默认为空，不包含真实敏感值。

## 16. 当前建议

在正式进入开发前，建议继续细化以下内容：

1. 细化会签规则、驳回规则和项目经理最终确认动作的页面交互。
2. 细化 Gitea 提交规范和凭证管理方式。
3. 细化敏感数据脱敏规则、审计日志字段和留存周期。
4. 细化后端代码结构规范、模板版本和生成校验规则。
5. 细化基础编译检查、启动验证、失败处理和交付阻断规则。
6. 明确第一阶段必须生成哪些产物，以及每个产物的确认人。
