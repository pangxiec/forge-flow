# ForgeFlow 审批流规则

## 1. 目标

审批流用于保障 ForgeFlow 从需求到 Git 交付的每个关键节点都经过明确确认，避免 AI 自动生成内容未经人工审核直接进入下一阶段。

审批流必须支持：

* 多角色确认
* 会签
* 或签
* 驳回
* 重新生成
* 版本冻结
* 项目经理最终确认
* 审计追踪
* 交付阻断

---

## 2. 核心原则

```text
AI 负责生成
人负责确认
流程负责阻断
Git 负责留痕
审计负责追责
```

任何关键产物未经确认，不允许进入下一阶段。

---

## 3. 角色定义

第一阶段支持以下角色：

```text
PROJECT_MANAGER     项目经理
PRODUCT_MANAGER     产品经理
BUSINESS_OWNER      业务方
ARCHITECT           架构师
FRONTEND_LEAD       前端负责人
BACKEND_LEAD        后端负责人
TEST_LEAD           测试负责人
DEVELOPER           开发人员
ADMIN               管理员
```

---

## 4. 审批动作

```text
APPROVE      同意
REJECT       驳回
COMMENT      评论
REGENERATE   要求 AI 重新生成
TRANSFER     转交
FINAL_CONFIRM 项目经理最终确认
```

---

## 7. 节点规则明细

## 7.1 需求确认节点

### 参与角色

```text
PRODUCT_MANAGER
BUSINESS_OWNER
```

### 会签规则

```text
产品经理必须同意
业务方必须同意
```

### 可执行动作

```text
APPROVE
REJECT
COMMENT
REGENERATE
```

### 通过条件

```text
产品经理 APPROVE
且
业务方 APPROVE
```

### 驳回规则

驳回后回到：

```text
需求补充 / AI重新分析需求
```

---

## 7.2 PRD 确认节点

### 参与角色

```text
PRODUCT_MANAGER
BUSINESS_OWNER
TEST_LEAD
```

### 会签规则

```text
产品经理必须同意
业务方必须同意
测试负责人必须同意
```

### 测试负责人确认内容

```text
验收标准是否明确
边界条件是否明确
异常场景是否明确
测试范围是否明确
```

### 通过条件

```text

```

### 驳回规则

驳回后回到：

```text
AI重新生成PRD
```

或：

```text
产品经理手动修改PRD
```

---

## 7.3 PRD 冻结节点

### 执行角色

```text
PROJECT_MANAGER
```

### 前置条件

```text
PRD确认节点已通过
所有未关闭澄清问题必须处理完成
PRD版本号已生成
```

### 冻结动作

```text
生成 PRD.md
生成 structured-prd.json
生成 acceptance-criteria.md
生成 questions.md
```

### 冻结规则

冻结后：

```text
当前 PRD 版本不可修改
如需修改，必须创建新版本
后续 Agent 必须基于冻结版本生成
```

---

## 7.4 原型确认节点

### 参与角色


```text

```

### 产品经理确认内容

```text

```

### 业务方确认内容

```text

按钮、状态、字段是否符合业务认知
```

### 前端负责人确认内容

```text
页面结构是否可实现
组件复杂度是否可控
交互状态是否明确
```

### 后端负责人确认内容

```text
页面所需接口是否明确
字段来源是否明确
状态流转是否明确
```

### 驳回规则

驳回后回到：

```text
AI重新生成原型
```

或：

```text
产品经理调整原型说明
```

## 8. 会签与或签规则

## 8.1 会签

所有指定角色必须审批通过。

示例：

```text
产品经理 + 业务方 + 测试负责人
```

适用于：

```text
PRD确认
原型确认
技术方案确认
开发包确认
```

## 8.2 或签

指定角色中任意一个通过即可。

示例：

```text
任意一个管理员
```

适用于：

```text
异常授权
重新触发失败任务
手动跳过非关键节点
```

## 8.3 项目经理确认

项目经理不替代其他角色审批。

项目经理只负责：

```text
冻结版本
最终确认
异常放行
流程终止
```

---

## 9. 驳回规则

驳回必须填写原因。

驳回原因必须包含：

```text
问题描述
影响范围
建议修改方向
是否需要 AI 重新生成
```

驳回后允许两种方式：

```text
1. 人工修改后重新提交
2. AI重新生成后重新提交
```

驳回记录必须写入：

```text
approval_record
audit_log
```

---

## 10. 重新生成规则

允许重新生成的节点：

```text
AI分析需求
AI生成PRD
AI生成原型
AI生成技术方案
AI生成开发包
```

重新生成时必须：

```text
创建新的 Agent Task
保留旧版本
生成新版本号
记录重新生成原因
记录触发人
```

禁止覆盖冻结版本。

---

## 11. 版本冻结规则

以下产物必须支持冻结：

```text
PRD
原型
技术方案
数据库设计
OpenAPI文档
开发包
```

冻结后：

```text
不可直接修改
不可覆盖
不可删除
只能创建新版本
所有下游产物必须记录来源版本
```

示例：

```text
代码包 v1.0 来源：
PRD v1.0
原型 v1.0
数据库设计 v1.0
OpenAPI v1.0
```

---

## 12. 流程阻断规则

以下情况必须阻断进入下一步：

```text
存在未确认审批
存在未关闭澄清问题
PRD未冻结
原型未冻结
技术方案未确认
基础编译失败
启动验证失败
敏感信息扫描失败
Git推送失败
审计记录缺失
```

阻断后状态：

```text
BLOCKED
```

必须人工处理后继续。

---

## 13. 审批超时规则

第一阶段可使用简单规则：

```text
审批超过 24 小时未处理：提醒审批人
审批超过 48 小时未处理：提醒项目经理
审批超过 72 小时未处理：标记为超时
```

通知方式：

```text
站内通知
邮件
企业微信 / 飞书 / 钉钉
```

## 16. 第一阶段必须实现的审批能力

MVP 必须实现：

```text
1. 节点审批
2. 多角色会签
3. 驳回
4. 评论
5. 重新生成
6. PRD冻结
7. 原型冻结
8. 项目经理最终确认
9. 审批日志
10. 流程阻断
```

第一阶段暂不强制实现：

```text
复杂条件流
动态审批人
加签
转签
委托审批
审批代理
```

## 18. 总结

ForgeFlow 审批流的核心不是简单“同意/拒绝”，而是确保 AI 生成的每一个关键产物都经过人类确认、版本冻结和审计留痕。

第一阶段审批流必须保证主链路稳定：

```text
需求确认
↓
PRD确认与冻结
↓
原型确认与冻结
↓
技术方案确认
↓
开发包确认
↓
Git交付确认
↓
项目经理最终确认
```

只有这条链路稳定，ForgeFlow 才能从 Demo 变成真正的企业级 AI 全链路研发平台。
